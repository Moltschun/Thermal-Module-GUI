# Thermal-Module-GUI

**Профессиональный графический интерфейс для управления промышленными тепловизионными модулями.**

![Python](https://img.shields.io/badge/Python-3.10%2B-blue) ![Qt](https://img.shields.io/badge/PySide-6-green) ![OpenCV](https://img.shields.io/badge/OpenCV-4.x-red) ![Status](https://img.shields.io/badge/Status-Active-brightgreen)

## О проекте

**Thermal-Module-GUI** — это высокопроизводительное приложение для захвата, визуализации и записи данных с тепловизионных камер высокого разрешения.

Проект разработан для решения проблем высокой задержки (latency) и артефактов буферизации при работе с потоковым видео Full HD (1080p) в среде Python/Qt. Архитектура приложения построена на модульном принципе, обеспечивая стабильную работу с промышленными интерфейсами Gigabit Ethernet / RTSP.

---

## Поддерживаемое оборудование

Проект включает драйверы для двух типов промышленных модулей:

| Производитель | Модель | Интерфейс | Протокол | Примечание |
|---|---|---|---|---|
| **Pergam / Dali** | Axion Uncooled HD 1920 | Ethernet (RJ45) | RTSP / ONVIF | Поддержка RTSP-ссылок DALI Web API |

> **Особенность Pergam:** Реализован обход проблемы авторизации ONVIF (рассинхронизация времени) и поддержка нестандартных портов (8000/554).

## В ходе разработки и анализа документации были выявлены ключевые характеристики модуля, определяющие архитектуру ПО:

**1. True HD Thermal Sensor (1920x1080)**
В отличие от большинства аналогов, использующих апскейл (программное растягивание), модуль Axion HD оснащен физической микроболометрической матрицей (FPA) с реальным разрешением 1920x1080.

Поток данных: ~2.1 миллиона пикселей на кадр.

**Нагрузка: В 7 раз выше**, чем у стандартных тепловизоров (640x480).

**2. Аппаратный предел 25 FPS**
Из-за высокой плотности пикселей пропускная способность шины данных сенсора и DSP-процессора ограничена частотой 25 Гц.

50 Гц в спецификации: Относится к внутренней частоте сканирования матрицы (Internal Clock), но выходной поток (RTSP) аппаратно прореживается до 25 кадров/сек для стабильности.

**3. Стратегия низкой задержки (Low Latency)**
Для минимизации буферизации тяжелого HD-потока применена гибридная стратегия:

Transport: Форсирование rtsp_transport=udp для исключения TCP-очередей.

Driver: Установка CAP_PROP_BUFFERSIZE = 0 на уровне драйвера OpenCV.

Flags: Использование флагов FFMPEG fflags=nobuffer|flags=low_delay.

## Ключевые возможности

### 1. Zero-Copy Rendering
Вместо стандартной медленной конвертации Base64 -> QML, используется кастомный QQuickImageProvider. Данные из буфера OpenCV передаются в видеопамять по ссылке, минуя лишнее копирование. Это обеспечивает плавную отрисовку без нагрузки на сборщик мусора (GC).

### 2. Асинхронная запись RAW
Реализован модуль FrameRecorder:

Буферизация: Кадры мгновенно сохраняются в очередь (RAM).

Фоновый сброс: Отдельный поток (Daemon Thread) сохраняет данные на диск в формате TIFF без сжатия, исключая "фризы" интерфейса во время записи.

### 3. Цифровая обработка (DSP)
Программное усиление яркости (Digital Gain) в реальном времени.

Корректная конвертация цветовых пространств BGR -> RGB для Qt.

## Архитектура
Проект использует паттерн MVC (Model-View-Controller), адаптированный под Qt:

* **View (main.qml)**: Декларативный интерфейс на QtQuick. Не содержит бизнес-логики. Получает видеопоток через протокол image://live.

Controller (AxionController.py):

* Управляет рабочим потоком (QThread).

* Обрабатывает сигналы от UI (кнопки записи, слайдеры).

* Реализует логику записи на диск.

* Worker (AxionWorker):

* Бесконечный цикл захвата (cv2.VideoCapture).

* Математическая обработка кадров (Numpy).

* Конвертация цветовых пространств (BGR -> RGB).
